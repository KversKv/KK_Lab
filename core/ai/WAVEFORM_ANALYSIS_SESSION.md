# 波形分析会话总结（电流波形特征提取 · 本地 LLM 数据传递优化）

> 范围：以 `tests/test.csv`（F1-A-I1 电流波形，5MB / 244,141 点 / 50kHz / 5s）为样例，复盘"海量原始数据 → AI 可消化结论"的完整链路；沉淀本次分析结论，并给出本地大模型在数据传递/结果表现上的优化规划。
> 关联事实源：`core/ai/providers/waveform_provider.py`（三层摘要实现）· `core/ai/schemas.py`（`WaveformStat`/`WaveformDigest`）· `core/ai/context_budget.py`（上下文预算）· `core/ai/DEBUG_WORKBENCH.md`（trace/replay/eval 闭环）。

## 0. 一句话结论

**AI 表现好的根因不是"能读懂 5MB"，而是"让本机 numpy 消化原始数据、AI 只消化代码吐出的几十行结论/JSON"。** 本项目 `waveform_provider.py` 已落地这套"三层摘要"范式，方向正确，需推广到本地 LLM 全链路。

## 1. 本次分析结论（tests/test.csv）

### 1.1 采集参数

| 项 | 值 |
|---|---|
| 通道 | F1-A-I1（电流） |
| 采样点数 | 244,141 |
| 采样间隔 dt | 20 µs（采样率 **50 kHz**） |
| 总时长 | 5.000 s |
| 电流范围 | −0.014 A ~ 33.27 A |
| 均值 / 标准差 | 1.558 A / 0.637 A |

### 1.2 波形行为：单次"上电 → 满载 → 稳态"启动序列（非周期脉冲串）

| 阶段 | 时间 | 特征 | 含义 |
|---|---|---|---|
| ① 静默 | 0 ~ 0.36 s | ≈ 3 mA | 未上电 / 休眠 |
| ② 上电涌流 | ≈ 0.36 s | 单点 16~24 A | inrush 冲击 |
| ③ 高负载初始化 | 0.41 ~ 1.0 s | 均值 2.5→2.84 A | 加载 / 初始化 |
| ④ 二次涌流 | ≈ 0.713 s | 单点 **33.27 A** | 过程二次浪涌 |
| ⑤ 稳态 | 1.07 s 后 | 稳定 **≈ 1.52 A**，纹波峰 ~2.0 A | 正常工作电流 |

### 1.3 0.40s ~ 0.42s 局部：4 次等间隔浪涌脉冲

| 脉冲 | 起始 | 峰值 | 宽度 |
|---|---|---|---|
| #1 | 0.409068 s | 16.82 A | ~140 µs |
| #2 | 0.411320 s | 16.80 A | ~140 µs |
| #3 | 0.413573 s | 23.83 A | ~100 µs |
| #4 | 0.415826 s | 14.87 A | ~60 µs |

- 间隔 2.253 / 2.233 / 2.232 ms → 平均周期 **2.24 ms ≈ 446 Hz**。
- 每脉冲仅 3~7 个采样点；脉冲前（0.400~0.409s）max 仅 0.22 A，无更早脉冲。
- **可信度边界**：50kHz 下尖峰只有 3~7 点，峰值（尤其 33A/23.8A 单点）可能被低估，需更高采样率才能准确测浪涌。

## 2. 操作流程复盘：原始数据是怎样被"压缩"喂给 AI 的

### 2.1 核心原则

原始 244,141 点（5MB）**从未进入 AI 上下文**。AI 接收的只有"代码运行后的结论文本/JSON"。

| 步骤 | 工具 | AI 实际接收量 | 作用 |
|---|---|---|---|
| 看格式 | 读前 30 行 | ~0.6 KB | 确认列名 `Time(s),F1-A-I1` 与数值格式 |
| 看规模 | 数行数 | 1 个整数 | 不读内容 |
| 提特征 | **写 Python → 运行 → 读 stdout** | ~2 KB 结论 | numpy 在本机算，AI 只读 print 结果 |

### 2.2 特征是"算法定义"的，不是"观察"的

```
采样率   = 1 / median(diff(t))
整体形态 = 50ms 窗口聚合（244141 点 → ~100 行 profile，降维）
脉冲数   = 阈值 + 上穿沿计数（diff(above)==1 的次数）
可信度   = 交叉验证（两种阈值得同一结果才下结论）
```

### 2.3 第二次（0.40~0.42s）：第一次范式的"放大镜"版

```
全局 244141 点
   │  numpy 切片 (t>=0.40)&(t<=0.42)   ← 在本机执行，AI 不接触这 976 点
   ▼
窗口 976 点
   │  阈值 5A + 上穿沿计数
   ▼
4 个脉冲事件 + 间隔统计               ← AI 仅接收这几行结果
```

**踩坑记录**：首版脚本曾把 976 行原始值全 `print`，终端被截断反而看不全。改为只 print "4 个脉冲事件"后干净可靠 —— **这正是本地 LLM 常犯的"灌数据而非灌结论"错误的缩影**。

## 3. 本地大模型"数据传递/结果表现差"的根因与优化

### 3.1 根因（反模式）

> ❌ 把 CSV/大段数据直接塞进 prompt 喂给模型。

- **上下文爆炸**：5MB ≈ 百万级 token，本地模型 4K~32K 上下文装不下 → 截断丢信息。
- **LLM 不会算数**：对 24 万浮点做统计是脑补 → 峰值/计数幻觉。
- **token 即算力**：灌越多越慢越卡。

### 3.2 正确范式：模型当"指挥官"，不当"计算器"

```
本地 LLM = 翻译意图 + 解读结论（不碰原始数据）
   用户问 "有几次脉冲"
        │ ① LLM 选择/生成分析调用（Tool / Function Call）
        ▼
   本机 Python(numpy) 消化 24 万点
        │ ② 仅回传几十行结论（JSON）
        ▼
   LLM 解析 JSON → ③ 组织人话回答
```

### 3.3 优化清单（按收益排序）

1. **绝不把原始数据进 prompt**：数据放磁盘，代码读磁盘，模型只读代码输出。
2. **给模型配工具而非数据**：预置 `get_signal_stats / count_pulses / downsample_profile` 等函数，模型只决定调谁、传什么参。
3. **降维后再给模型看**：窗口聚合到 ~100~500 点（对应 `waveform_provider` 第 2 层 LTTB）。
4. **结构化输出 JSON**：模型解析 `{"pulse_count":4,...}` 比解析散文可靠。
5. **结论带可信度边界**：脚本顺带输出"每脉冲采样点数"，模型自动加免责说明。
6. **多步收敛**：先全局降维看大概，再对小区间精算（对应第 3 层 `slice_window`），每步只传小结果。

## 6. 当前本地代码流程（实读 `waveform_provider.py` / `n6705c_datalog_ui.py` 后）

### 6.1 数据从哪来 → 怎么进内存

事实源：`ui/pages/n6705c_power_analyzer/n6705c_datalog_ui.py`。

```
仪器 datalog / 导入 CSV
        │  解析为 Python list（非 numpy）
        ▼
all_data = { label: {"time": [float, ...], "values": [float, ...]} }   ← 全量驻留内存
        │  UI 线程持有 self.datalog_data
        ▼
build_waveform_digest(x_range, marker)   （UI 线程，n6705c_datalog_ui.py:5312）
```

- 原始点是**全量常驻 Python list**；mock 路径有 `total_points = min(total_points, 50000)` 的硬上限保护，真机/导入路径无此上限。

### 6.2 三层摘要管线（事实源 `core/ai/providers/waveform_provider.py`）

```
第 1 层 统计摘要  build_digest()
   _statistics()         min/max/avg/std（纯 Python sum 循环，O(n)）
   _detect_anomalies()   |v-mean|>σ·std 的异常点，限量 20
   _cluster_spike_events() 超阈点按 gap_s 邻近聚簇 → 尖峰事件（解决"一簇多点误判多脉冲"）
   _detect_steady_segments() 局部波动 < 0.5·std 的稳态段，限量 5
   _pick_scale()         按量级选 SI 前缀（mA/A 统一量纲）
        │
第 2 层 形状降采样  lttb_downsample()
   Largest-Triangle-Three-Buckets，默认降到 ~1500 点，保留视觉峰谷（纯 Python for）
        │
第 3 层 按需放大  build_window_digest() / slice_window()
   slice_channel_fast()  等步长 O(1) / 非等步长 bisect O(log n) 定位区间，禁 O(n) 全遍历
   超 max_points 仍再 LTTB 压一遍
```

### 6.3 产物如何喂给 LLM

```
WaveformDigest（schemas.py：stats + downsampled + note + window + marker_segment）
        │  to_dict() → JSON
        ▼
context_builder 组装 → context_budget.py 按"当前模型窗口"裁剪（tiktoken/启发式估 token）
        ▼
本地 LLM 只收到 JSON 摘要 + ~1500 点降采样曲线（原始百万点不进上下文）
        ▲
        │ AI 看完摘要后，可调用 action get_waveform_window(label,t0,t1) 主动放大
        └─ drill-down 回到第 3 层
```

**结论：你的本地流程已经实现了"原始点不进上下文 + 三层降维 + AI 主动 drill-down"的正确范式，且切片定位做了 O(1)/O(log n) 优化。**

## 7. 两种套路对比（我的临时脚本 vs 你的 provider 管线）

| 维度 | 我的临时脚本套路 | 你的 `waveform_provider` 管线 |
|---|---|---|
| 触发方式 | 即兴写 `.py` → 跑 → 读 stdout | 工程化函数 + 注册为 AI action，可被 LLM 自动调用 |
| 数据载体 | numpy 数组（`genfromtxt`） | Python list（标量循环） |
| 计算性能 | 向量化，24 万点毫秒级 | 纯 Python O(n)，24 万点可接受，更大量级会慢 |
| 降维方式 | 50ms 窗口均值聚合（简单） | LTTB 保峰谷（更利于 AI/绘图看形状） |
| 局部放大 | 重新切片 + 重算（每次全 numpy） | `slice_channel_fast` O(1)/O(log n)，不重新采数 |
| 结构化输出 | print 散文文本 | `WaveformDigest.to_dict()` → JSON（可校验、可回归） |
| token 控制 | 无（靠我手动只 print 结论） | `context_budget` 按模型窗口自动裁剪 |
| 可复现/可回归 | 一次性脚本 | 可接 `eval_runner` 做 golden set 回归 |
| 灵活性 | 高（任意临时分析、任意阈值） | 受限于已实现的算子（脉冲检测/间隔统计尚未成标准算子） |

**优缺点小结**

- **我的套路**：胜在*灵活、向量化快、即兴*；输给*不可复现、输出非结构化、无 token 治理、不可被 LLM 自动调用*。适合"探索性一次分析"。
- **你的管线**：胜在*工程化、结构化 JSON、token 受控、可被 AI 调用、可回归、切片高效*；短板是*纯 Python 标量计算在大数据下吃亏*、*算子覆盖面还窄*（缺等间隔脉冲串的周期/重复率检测、缺可信度边界字段）。适合"产品化、可持续"。

**最佳实践 = 两者融合**：保留你的三层管线与 action 架构，把内部标量循环换成 numpy 向量化，并把我临时脚本里的"脉冲计数 + 间隔/重复率 + 可信度边界"沉淀为标准算子填进 `spike_events`。

## 8. 大数据可扩展性：5MB 通用，150MB / 1GB 是否通用？

### 8.1 结论先行

| 数据规模 | 当前管线是否通用 | 主要瓶颈 |
|---|---|---|
| 5MB（~24 万点） | ✅ 通用，秒级 | 无 |
| 150MB（~700 万点） | ⚠️ 勉强，明显变慢/占内存 | 纯 Python O(n) 循环 + list 内存膨胀 |
| 1GB（~5000 万点） | ❌ 不通用，会卡死/OOM 风险 | 全量 list 驻留内存 + 多次 Python 遍历 + 解析耗时 |

### 8.2 三大瓶颈（按 1GB 量级）

1. **内存**：Python `list[float]` 每个元素 ~24–28 字节（对象 + 指针），1GB 原始 CSV 解析成 list 可膨胀到 **数 GB ~ 十几 GB**，极易 OOM。numpy `float64` 数组每点仅 8 字节，差一个数量级。
2. **CPU**：`_statistics` / `lttb_downsample` / `_detect_*` 全是 Python 级 for 循环，5000 万点要遍历多趟，**单趟就达秒~十秒级**，叠加多算子体验崩坏。
3. **解析与加载**：一次性把 1GB CSV 读进内存本身就慢且阻塞，UI 线程会卡死（即便在 worker 也占满内存）。

> 注：`slice_channel_fast` 的区间定位已是 O(1)/O(log n)，**不是瓶颈**；瓶颈在"全量驻留 + 全量遍历 + 解析"。

### 8.3 可扩展化改造路线（规划）

| 优先级 | 改造 | 收益 |
|---|---|---|
| P0 | provider 内部计算改 **numpy 向量化**（`np.mean/std`、布尔掩码找超阈、向量化 LTTB） | CPU 提速 10~100×，内存减少一个数量级 |
| P0 | 原始数据载体从 list 改 **numpy ndarray**（`all_data` 的 values 存 `np.ndarray`） | 内存 8 字节/点；切片零拷贝（视图） |
| P1 | 大文件用 **np.memmap / 分块解析**（不一次性进内存），或落地为 `.npy`/parquet 列存 | 1GB 不再全驻留，按需映射 |
| P1 | 统计摘要走 **分块/流式（chunked）聚合**（Welford 在线均值方差、分块 min/max） | 单趟、低内存、可处理 >内存 数据 |
| P2 | 第 2 层降采样改 **多分辨率金字塔/预降采样缓存**（min-max bin + LTTB） | 摘要 O(1) 取用，drill-down 才取高分辨率 |
| P2 | 解析/分析强制走 **QThread worker + 进度上报**，UI 不阻塞（遵分层铁律） | 大文件不卡 UI |
| P3 | 超大数据引入 **pyarrow/polars/duckdb** 做列式按需查询（仅在确有 GB 级需求时） | SQL 式窗口聚合，磁盘级可扩展 |

### 8.4 对 LLM 上下文的影响（关键洞察）

**好消息：无论原始数据是 5MB 还是 1GB，喂给 LLM 的 `WaveformDigest` 体积几乎不变**——统计摘要是固定几十个标量，LTTB 固定 ~1500 点，drill-down 固定 ~2500 点。即 **AI 侧天然可扩展，瓶颈只在"本机如何高效算出这个固定大小的摘要"**。这正印证 §3 的核心范式：把"消化大数据"的压力全部留在本机计算层，LLM 永远只吃恒定小摘要。

### 8.5 一句话

当前管线在 **架构上对大数据是通用的**（三层降维 + 恒定摘要 + drill-down），但在 **实现上对 150MB/1GB 不通用**（纯 Python list + 标量循环）。把计算层换成 numpy 向量化 + 数据载体换成 ndarray/memmap + 流式分块聚合，即可在不改变 AI 接口的前提下平滑扩展到 GB 级。

## 9. 事件感知特征提取：变更规划 · 实现计划 · 性能分析

### 9.1 要解决的问题（LTTB 的结构性盲区）

`lttb_downsample` 为"看整体趋势"而生，对 **稀疏窄脉冲结构性失明**：
- 1.28s interval BT 唤醒 @ 50kHz → 每 interval 64000 点，唤醒脉冲仅几十点；降到 ~1500 点时每桶 ~42667 点，唤醒峰**可能被丢、形状必丢**。
- `_cluster_spike_events` 用 `σ·std` 阈值，在"99% 睡眠基线"下 std 被压塌、`max_report=20` 截断、且**无周期/形态刻画**。

三种朴素方案的局限（已讨论）：
- **方案 A（min-max 保极值）**：受均匀桶密度限制，平稳段仍浪费预算，唤醒段密度不足。
- **方案 B（事件感知自适应密度）**：方向对，但必须**显式告知 AI 时间轴是非均匀采样**，否则 AI 误判密度。
- **方案 C（周期事件特征）**：假设周期性，**SOC 不规则外设唤醒不适用**，场景太专一。

### 9.2 选定方案：双引擎"事件感知"管线（按信号形态路由，不假设周期）

> ⚠️ **重大修正（经 §9.7 / §9.8 实测验证）**：原把 PELT 列为"可选增强"是错误的。实测证明 **STA-LTA 与 PELT 是互补的两类必需引擎**——前者只抓"窄尖峰/突发"，对"中等幅度平台/电平台阶"结构性失明；后者专抓"均值台阶"，二者缺一不可。功耗信号同时存在这两类形态（涌流尖峰 + RX/状态平台），故**双引擎是必需，不是可选**。

```
原始海量点（ndarray）
   │
   ① 形态路由（廉价 O(N) 预判：基线占比 / 峰均比 / 段时长分布）
   │      ├─ 窄尖峰/突发  →  STA-LTA 能量比 + MAD 绝对幅值闸门（自研·零依赖）
   │      └─ 平台/电平台阶 →  PELT 变点检测（均值代价，自研·零依赖；高精度可选 ruptures）
   │
   ② 段形态分类  对每个候选段贴标签：尖峰 / 平台 / 低谷 / 缓变
   │      └─ 规则：宽度 + 峰均比(peak/mean) + 均值层级
   │
   ③ 自适应降采样  平稳段 min-max 稀疏 + 事件段高密度保留（方案 B）
   │      └─ 同步产出 density_map（显式标注各区段采样密度）
   │      └─ 极值层用 M4/MinMaxLTTB 思想保证峰谷不丢（方案 A 升级版）
   │
   ④ 段落特征化  每段算 均值/峰值/峰均比/宽度/上升下降/积分电荷(µAh)/能量
   │      └─ 跨段聚合：间隔分布/形态聚类（仅作描述，不假设规律）
   │
   ⑤ 结构化 JSON  segments[]+labels + density_map + feature_table → 喂 LLM（恒定 token）
```

**算法来源（均有学术/工业背书，可离线、可向量化）**：
| 环节 | 算法 | 出处 / 库 | 选型理由 |
|---|---|---|---|
| 事件定位·尖峰 | **STA-LTA** 短/长时窗能量比 + **MAD** 闸门 | 地震学经典 | 专为"长静默+突发"设计；十几行自研、无依赖；MAD 抗睡眠基线塌陷 |
| 事件定位·平台 | **PELT** 变点检测（均值代价） | Killick 2012 / `ruptures` | 精确分割均值台阶；对 RX/电平/DVFS 切换不可替代；参数鲁棒 |
| 事件定位·在线 | **BOCPD** 贝叶斯在线变点 | Adams&MacKay 2007 | 流式/超长采集增量检测（可选） |
| 段形态分类 | 规则（宽度+峰均比+层级） | 自研 | 给每段贴 尖峰/平台/低谷 标签，避免误把尖峰当平台 |
| 保极值降采样 | **M4 / MinMaxLTTB** | Jugel VLDB2014 / `tsdownsample` | 像素级保证不丢峰谷；Rust 加速解 §8 性能 |
| 形态聚类(可选) | **Matrix Profile / FLOSS** | `stumpy` | 无监督找重复模式与区段边界 |

**分工**：算法不自造（STA-LTA / PELT 均可十几~几十行自研、零依赖）；自研"形态路由 + 段形态分类 + 段落业务特征 + density_map 标注 + JSON 装配"这一薄层，沉淀进 `waveform_provider.py`。高精度场景可选挂 `ruptures`/`tsdownsample`，缺包自动回退自研实现。

### 9.5 性能分析

**复杂度（设原始 N 点、事件数 E、目标点 M）**

| 步骤 | 复杂度 | 说明 |
|---|---|---|
| STA-LTA 事件定位 | O(N) 单趟（向量化滑动能量比，`np.cumsum` 前缀和实现 O(1) 窗口） | 无需排序，一趟扫描 |
| 自适应降采样 | O(N)（min-max 分箱）+ O(事件段点数) 高密度保留 | 平稳段 min-max，事件段近原样 |
| 段落特征 | O(Σ事件段点数) ≪ O(N) | 只在事件区间内算，睡眠段跳过 |
| **合计** | **O(N) 单趟主导** | 与现 LTTB 同阶，但信息密度更高 |

**纯 Python vs numpy 向量化实测预估（基于本会话 5MB/24万点经验外推）**

| 数据规模 | 纯 Python 标量循环 | numpy 向量化（目标实现） |
|---|---|---|
| 24 万点（5MB） | 数百 ms（可接受） | < 20 ms |
| 700 万点（150MB） | 10~30 s（明显卡顿） | 0.2~0.5 s |
| 5000 万点（1GB） | 分钟级 / 易 OOM | 1~3 s（配 ndarray 载体）；`tsdownsample` 更快 |

**内存**：沿用 §8 结论——ndarray 载体每点 8 字节，较 Python list（~24–28 字节/点）省一个数量级；STA-LTA 用前缀和数组，额外内存 O(N) 一份，1GB 数据可控。

**对 LLM 上下文的影响（关键）**：输出仍是**恒定小体积**——`segments`（每事件几十字节，限量上报+全量统计）+ `density_map`（几行）+ `downsampled`（保极值后约 1500~3000 点）。**无论 N=24万还是 5000万，喂 LLM 的 token 几乎不变**，瓶颈只在本机计算层（已被 numpy 向量化解决）。

**性能风险与对策**：
- STA-LTA 窗口参数（`sta_s`/`lta_s`）对不同采集率敏感 → 按 `sample_period_s` 自适应设默认，暴露可调参。
- 事件极多（如 60s × 高频唤醒 = 上千事件）→ `segments` 走"全量统计 + 异常/代表性事件限量上报"，避免 token 膨胀。
- 可选依赖缺失 → C8 自动回退自研实现，保证内网/离线可用、不破坏 `requirements.txt` 约束。

### 9.7 原型验证结果（STA-LTA + MAD 闸门，基于 tests/test.csv）

事实源：`tests/_stalta_tmp.py`（零依赖，仅 numpy；用 `np.cumsum` 前缀和实现 O(1) 滑窗能量比）。

**算法两步法（关键结论）**：单一手段都不够，必须组合——
1. **STA-LTA 能量比定位"哪里有突变"**：`STA=0.10ms / LTA=5.00ms`，触发 `on/off=4.0/1.5`，邻近 `gap=0.2ms` 合并 → 全程得 **31 个候选事件**（含 24 个睡眠段微扰动伪事件）。
2. **MAD 绝对幅值闸门裁定"突变够不够大"**：`floor = median + 6·MAD·1.4826 = 1.5077 + 6·0.0749·1.4826 ≈ 2.174 A` → **31 → 6 个有效事件**，伪事件全滤除。

**关键参数对比（实测）**：

| 配置 | 0.40~0.42s 簇数 | 副作用 | 结论 |
|---|---|---|---|
| `on=8.0` 高阈，无 MAD | **3** ❌ 漏 14.87A 弱簇 | 强簇 23.83A 抬高 LTA 遮蔽紧邻弱簇（LTA 污染） | 单纯调高阈值会漏弱事件 |
| `on=4.0` 低阈，无 MAD | **4** ✅ | 误报 24 个 0.02A 睡眠微扰 | 纯能量比无幅值概念 |
| `on=4.0` 低阈 **+ MAD 闸门** | **4** ✅ | 伪事件清零，全程仅 6 有效事件 | ✔ 最终方案：能量比保灵敏 + MAD 滤幅值 |

**全程 5s 自动提取结果（无预设窗口 / 无手工阈值）**：

| 事件 | 起止时间 (s) | 峰值 (A) | 宽度 (ms) | 电荷 (µAh) | 采样点 | 归属 |
|---|---|---|---|---|---|---|
| E01 | 0.409068~0.409252 | 16.82 | 0.184 | 0.445 | 10 | 0.41s 浪涌簇 #1 |
| E02 | 0.411341~0.411525 | 16.80 | 0.184 | 0.456 | 10 | 簇 #2 |
| E03 | 0.413594~0.413737 | 23.83 | 0.143 | 0.301 | 8 | 簇 #3 |
| E04 | 0.415846~0.415949 | 14.87 | 0.103 | 0.090 | 6 | 簇 #4 |
| E05 | 0.710308~0.710533 | 16.27 | 0.225 | 0.362 | 12 | 二次涌流前沿 |
| E06 | 0.712561~0.712724 | 33.27 | 0.163 | 0.528 | 9 | 二次涌流峰 |

- 0.40~0.42s 窗口内 **4 簇全分出**，峰值 16.82/16.80/23.83/14.87A 与 §1.3 逐一吻合。
- 上电涌流（§1.2 阶段②，~0.36s）被 §1.3 的 4 簇与 0.41s 高负载初始化覆盖；0.71s 二次涌流由 E05/E06 捕获（33.27A 与 §1.2 阶段④一致）。
- 每事件附 峰值/宽度/电荷，电荷=∫i·dt（`np.trapezoid`，新版 numpy 改名，已做兼容回退）。

**验证结论**：
1. **可行性确认**：STA-LTA 能在不预设窗口、不手工设 5A 阈值的前提下，从全程 5s 自动定位事件——这是固定阈值法做不到的（固定阈值需人先框窗口、先知基线）。
2. **"一次到位"需打引号**：裸 STA-LTA 对阈值敏感（高阈漏弱簇 / 低阈误报），**必须叠加 MAD 绝对幅值闸门**才能"既不漏弱簇又不误报睡眠扰动"。
3. **印证 §9.2 选型**：主引擎 = STA-LTA（定位"哪里有突变"，能量比，对相对变化敏感）+ MAD 闸门（裁定"突变够不够大"，绝对幅值，抗睡眠基线塌陷）。这正是对现有 `_cluster_spike_events` 用 `σ·std` 的关键修正——`σ·std` 在 99% 睡眠基线下 std 被压塌而失效，MAD 用中位数+绝对偏差天然抗这种极端不平衡分布。
4. **已知短板与对策**：STA-LTA 存在"强事件遮蔽紧邻弱事件"（LTA 污染），落地时需补"触发期冻结 LTA 更新"或保持低 on 阈 + MAD 兜底。

> 产物：`tests/_stalta_tmp.py`（零依赖原型，可作 §9.4 阶段 1 固化基线）。

### 9.8 双引擎铁证：PELT 暴露 E04 内部三次 RX（基于 tests/test.csv）

事实源：`tests/_rx_tmp.py`（原始点打印）+ `tests/_pelt_tmp.py`（自研 PELT 均值变点，零依赖）。

**背景**：§9.7 中 0.4154~0.4169s 仅被 STA-LTA+MAD 识别为"E04 单一尖峰（峰 14.87A / 6 点）"。实测告知此处实际有**三次 RX 行为**。验证当前算法能否暴露其内部子结构。

**STA-LTA+MAD 对 RX 的表现：❌ 结构性失明**
- 三次 RX 是 ~3.4~4.0A 的**中等幅度等高平台**（非窄尖峰）。MAD 闸门 floor=2.174A 把平台当背景滤除；STA-LTA 只对"突变沿"敏感、对"平台本体"无感。
- 仅 RX1 末尾恰好叠加的 14.87A 尖峰被偶然抓到（即 E04），RX2/RX3 纯平台**完全漏掉**。

**PELT 变点检测对 RX 的表现：✅ 完美**（pen=3/6/10 结果稳定，参数鲁棒，切出 8 段）：

| 段 | 时间 (s) | 均值 (A) | 峰值 (A) | 宽度 (ms) | 电荷 (µAh) | 形态标签 |
|---|---|---|---|---|---|---|
| S1 | 0.41542~0.41546 | 1.32 | 1.36 | — | 0.015 | 低谷 |
| **S2** | 0.41548~0.41581 | **3.43** | 3.82 | 0.327 | 0.305 | **RX1 平台** |
| S3 | 0.41583~0.41587 | 10.57 | **14.87** | 0.041 | 0.129 | 尖峰(峰均比1.4·窄)→叠于 RX1 |
| S4 | 0.41589~0.41597 | 1.52 | 1.80 | 0.082 | 0.033 | 低谷(RX1↔RX2 间隔) |
| **S5** | 0.41599~0.41630 | **3.57** | 3.77 | 0.307 | 0.298 | **RX2 平台** |
| S6 | 0.41632~0.41648 | 1.58 | 2.72 | 0.164 | 0.065 | 低谷(RX2↔RX3 间隔) |
| **S7** | 0.41650~0.41681 | **4.04** | 4.15 | 0.307 | 0.337 | **RX3 平台** |
| S8 | 0.41683~0.41689 | 1.39 | 1.79 | — | 0.022 | 回落 |

- **三次 RX（S2/S5/S7）干净提取**：均值 3.43/3.57/4.04A，宽度近乎相等 ~0.31~0.33ms，电荷 0.305/0.298/0.337µAh，间隔规整 → 典型周期性 RX 指纹。
- PELT 把尖峰段（S3）也切成独立段——印证需 **"PELT 切分 + 段形态分类"两段式**：PELT 负责把信号切成均质段，分类器（宽度+峰均比+层级）再贴 平台/尖峰/低谷 标签，避免简单 mean 阈值把尖峰误算成 RX。

**三类算法在该窗口的对比（决定性铁证）**：

| 算法 | 3 个 RX 平台 | E04 尖峰 | 定性 |
|---|---|---|---|
| LTTB 降采样 | ❌ 失明 | ❌ 失明 | 趋势算法，稀疏窄/中幅特征全丢 |
| STA-LTA + MAD | ❌ 失明 | ✅ 完美 | 尖峰/突发专精，平台盲区 |
| **PELT 变点检测** | ✅ **完美** | ✅ 切为独立段 | 电平/平台专精，尖峰也能切出 |

**结论（升级依据）**：
1. **单一算法不可能通吃**：功耗信号同时含"窄尖峰（涌流/唤醒）"与"中幅平台（RX/状态/DVFS）"两类形态，分属 STA-LTA 与 PELT 的主场，互为盲区。
2. **PELT 从"可选增强"升级为"必需引擎"**（§9.2 已修正），与 STA-LTA 并列双引擎，按形态路由。
3. **drill-down 必须换引擎**：当 AI 对一个已识别的"尖峰事件"窗口请求放大时，窗内子结构（如 RX 平台串）应自动改用 PELT 重扫，而非沿用 STA-LTA。
4. **需新增"段形态分类"层**（C11）：PELT 只切分、不分类；标签由 宽度/峰均比/均值层级 规则判定。

> 产物：`tests/_rx_tmp.py`、`tests/_pelt_tmp.py`（零依赖原型，PELT 自研均值变点 + 段特征）。
